## Resolution
<!-- OVERWRITE as understanding evolves -->

root_cause: "Track 10's right-edge width spike (0.228 -> 0.147 -> 0.800 mm at grid indices 397/398/399, x=99.5/99.7/99.9mm) is caused by a degenerate zero-smoothing condition in `nan_savgol()` (src/targets.py:144-162), not by a masking/gap-logic bug. Track 10 has a legitimately A2-invalidated 3-column gap at indices 394-396 (x=98.9-99.3mm) — its largest half-max-threshold run touches y=0 (the first measured y-pixel), correctly triggering the user-ratified A2 rule ('a run touching the measured-strip boundary is invalid'). This gap sits close enough to the true right crop-grid edge (index 399 = last of 400 grid points) that `nan_savgol`'s local window (ideal size 5, half-width 2) can only ever find exactly 3 finite neighbors for ALL THREE of indices 397, 398, and 399 — not just the single true-edge point. Because `nan_savgol`'s polynomial-degree formula is `degree = min(SG_POLYORDER=2, finite_count-1)`, a 3-point window yields degree=2, which is an EXACT/interpolating fit (zero residual degrees of freedom) rather than a genuine smoothing/averaging fit. Verified numerically: the 'smoothed' output is bit-identical to the raw (pre-smoothing) per-column half-max-edge estimate at indices 393, 397, 398, and 399 for Track 10. Track 10's raw per-column half-max estimates in this stretch are noisy (isolated single-column crossings feeding a known general-jaggedness issue, sibling gap G-01-2), so with zero smoothing applied, that raw noise passes straight through into the persisted/plotted output, producing the visually alarming terminal spike. Track 14 is spared this specific 3-point degeneracy (only its single true-last grid point, index 399, is similarly unsmoothed — the same as every track, and already an expected/documented D-12 'shrinking window near crop edges' behavior) purely because Track 14's preceding A2 gap happens to be 1 column shorter (2 columns vs Track 10's 3), leaving a full/adequate finite window at indices 397 and 398. This is a real, reproducible implementation gap in nan_savgol's window-size handling (it silently disables smoothing whenever a window shrinks to exactly SG_POLYORDER+1 finite points, without flagging the position as reduced-confidence or invalid) combined with Track 10 specifically having a data-scarce region (correctly masked per ratified A2) positioned unluckily close to the crop boundary. It is best characterized as BOTH: (a) genuine sparse-data-at-the-edge circumstance specific to Track 10 (not fabricated, not a masking error), amplified by (b) a real, generalizable code defect — nan_savgol provides zero noise suppression whenever finite-window-size <= SG_POLYORDER+1, which was not called out in D-08/D-09/D-12's documented smoothing design and silently defeats the 'preserve local peaks/slopes better' SG intent at exactly the positions that need damping most."
