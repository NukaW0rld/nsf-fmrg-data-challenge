## Root causes (three concurrent mechanisms)

**Mechanism A — merge-before-clip ordering (new defect introduced by the 01-14/G-01-5 fix; dominant driver of worsened coverage/fragmentation counts).**
`halfmax_edges` runs `merge_adjacent_runs` (`src/targets.py:168-183`) *before* the D-01/D-03 boundary-clip-exclusion filter (`src/targets.py:200-213`). When a raw above-threshold run already touches a y-strip edge (correctly excluded on its own per D-01/D-03) sits within `MAX_RUN_MERGE_GAP_PIXELS=10`px of another otherwise-valid run, the merge fuses them into one span that *also* touches the edge — so the entire fused span, including the legitimate interior run, is discarded. Measured: of all `no_candidates` invalidations, 9/13 (69%) on track 8, 56/86 (65%) on track 10, 9/10 (90%) on track 14, 37/48 (77%) on track 21 are directly attributable to this mechanism (pre-merge candidate existed, post-merge it vanished, and 100% of those had a pre-existing edge-touching raw run). A merge-gap parameter sweep (0/5/10/15/20/30px) shows swallow count increases monotonically with gap size (e.g. track 10: 0→41→56→64→73→93) — confirming this is a structural ordering defect, not a mistuned constant. Severity correlates with each track's chronic y-strip-edge threshold-exceedance in the detrended residual (track 10 y-index-0: 94.8% of columns above threshold; track 21 y-index-479: 91.4%; tracks 8/14 far lower). This also corrects 01-14-ORDERING-OUTCOME.md's own stated interpretation (which attributed the worsening to the plausibility gate) — the gate cannot zero out a non-empty candidate list by construction, since the max-length candidate always satisfies its own ratio; every empirical zero-candidate case traced to Mechanism A instead.

**Mechanism B — same-column-relative gate provides no protection for single-candidate columns (residual gap in the 01-14 fix).**
`MIN_TRACKED_LENGTH_RATIO` only helps when a column has both a large and a small candidate to compare. When exactly one (small/spurious) candidate exists, the gate is trivially satisfied. Measured fraction of tracked selections occurring in single-candidate columns (gate structurally moot): track 8 21.7%, track 10 72.4%, track 14 14.0%, track 21 37.6%. Directly traced at track 8 x=24.1/24.5/24.7/25.5mm (single candidates of length 1-32 native px selected with no alternative, producing 1-2-column width collapses between correctly-tracked ~1.0-1.1mm neighbors). Same failure class as G-01-2/G-01-5 but now bounded to 1-2 columns instead of 8+, trading fewer long locks for more frequent short blips — mechanically increasing the contiguous-run fragmentation count, matching both 01-14-ORDERING-OUTCOME.md's "contiguous run counts got worse on tracks 8/10/21" and the G-01-6 "frequent EKG-like jumps" report.

**Mechanism C — greedy nearest-to-previous_center selection among multiple simultaneously-plausible candidates (explicitly deferred by 01-14, confirmed still active).**
When a column has multiple candidates all ≥50% of the max length, at very different y-locations, proximity-only selection can hop between distinct real features with no protection beyond raw distance to a potentially-already-wrong anchor. Measured at track 21's tail (x=99.1-99.9mm): raw tracked y-center hops 0.27→0.14→**1.52**→0.19→0.53mm across 5 consecutive columns, each with 3-7 merged candidates. This is the DP/Viterbi joint-tracker gap 01-14-PLAN.md named and explicitly deferred as out of scope.

## Crop-edge-specific findings

- **Right edge (track 10 island, track 21 terminal drop):** Same Mechanisms A and C, at their most acute — domain edges are exactly where each track's chronic y-strip-edge elevation is most active, and native sampling density is unchanged there (confirmed 50 native columns per 0.2mm bin at every track-10 crop-edge x-position — not a sparse-data issue). Whether an edge column survives is effectively a coin-flip on a handful of native pixels of separation, producing the observed alternating valid/gap pattern rather than a clean run or clean gap.
- Track 21's terminal drop (0.356→0.340→0.107mm at x=99.5/99.7/99.9mm, the last 3 of 400 grid points) is compounded by `nan_savgol` (`src/targets.py:259-277`) using a truncated window at the array's last 1-2 indices (`hi = min(len(v), i+half+1)` shrinks the nominal 5-point window to 3 points at i=398-399) — less noise-averaging exactly where Mechanism C's instability needs it most.
- **Left edge (all four tracks, ~x=20.0-22.3mm unassessable):** Different, non-algorithmic mechanism — genuine native-data unavailability at the scan/crop start (measured native `x_actual_mm` minimums: track 10=20.0016mm, track 8=20.264mm, track 21=20.726mm, track 14=21.463mm). Traces entirely to the pre-existing, already-locked `no_bin` (`MIN_COLUMNS_PER_BIN=10`) and `big_gap` (`fill_small_gaps`'s `MAX_GAP_PIXELS=10`) paths — not Mechanisms A/B/C. Matches the reviewer's own "unassessable" (not "implausible") framing; expected, correctly-disclosed gap-handling behavior, not a defect.

## Suggested fix direction

1. **Mechanism A (tractable, scoped):** Apply the D-01/D-03 clip-exclusion test to each raw run *before* merging; merge only among already-non-edge-touching runs (or equivalently, forbid `merge_adjacent_runs` from combining across any run that itself touches the y-strip boundary). Validated in this investigation (throwaway analysis, not applied): recovers all 111 swallowed columns across all four tracks with no loss of the noise-fragmentation-merging benefit for genuinely interior fragmented beads.
2. **Mechanism B (tractable, scoped):** Add a distance-from-recent-tracked-history dimension to the gate (not just same-column relative size), so a lone candidate that is both far from `previous_center` and small relative to recent tracked width can be rejected even with no same-column competitor.
3. **Mechanism C (honest limitation, per HONEST-OUTCOME GUARD):** No new tractable fix beyond what 01-14 already considered and correctly deferred (full DP/Viterbi joint sequence tracker) was found. Its cost concentrates at domain far-edges and on tracks 10/21 specifically (the tracks with chronic single-side edge elevation) — it is the direct cause of the still-open crop-edge complaint, not a diffuse low-priority residual, but resolving it fully requires the sequence-global method already named and scoped out.
4. **Left-edge bands:** No fix indicated; recommend accepting as expected, correctly-disclosed native-data-availability behavior.
